# Markdown 解析与编辑方案

## 方案比较

### 三态视图：富文本 / 源码 / 预览

优点是能力边界清楚，源码编辑和只读预览都容易解释。缺点是便签场景里模式切换成本偏高，用户容易不确定当前修改会写入哪个视图；源码模式还需要频繁把 Markdown 解析回编辑器文档，和图文混排、任务列表状态存在同步压力。

### 纯 Markdown 源码 + 预览

优点是 Markdown 语义最直接，适合长文和开发者工作流。缺点是便签应用常见的图片拖拽、富文本快捷工具、任务列表勾选会变弱；大文档下预览和编辑区双栏同步也更容易产生滚动、光标和性能问题。

### Typora 式实时渲染

优点是只有一个编辑面，用户输入 `# `、`- `、`> ` 等 Markdown 语法时立即转成结构化文档；图文混排、工具栏、快捷键、附件迁移和历史版本仍由 TipTap/ProseMirror 的文档模型承载。缺点是它不是逐字保留 Markdown 源码的编辑器，Markdown 导出会由序列化器生成规范化文本。

## 采用方案

采用 Typora 式实时渲染：

- 编辑器只保留一个 TipTap 编辑面，移除源码/预览切换。
- `StarterKit` 保留 Markdown input rules，输入时实时生成标题、列表、引用、代码块等节点。
- Markdown 导入使用 `setContent(markdown, { contentType: "markdown" })`。
- 复制与导出使用 `editor.getMarkdown()`，由 `@tiptap/markdown` 统一序列化。

这个方案更符合 QuickNote 的定位：快速记录、本地优先、图文混排，同时避免三态视图造成的同步和认知成本。
