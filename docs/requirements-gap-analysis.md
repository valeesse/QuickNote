# QuickNote 需求方案与差距分析

## 目标定位

QuickNote 应优先做成本地优先、低延迟、跨平台的高性能便签应用。核心体验不是“大文档写作”，而是随时呼出、快速记录、稳定保存、可靠搜索，并自然支持图文混排和 Markdown 风格输入。

## 推荐 MVP

- 跨平台桌面：Tauri + React，数据默认保存在本机应用目录。
- 编辑体验：TipTap 富文本编辑器，支持标题、列表、任务列表、引用、代码块、高亮、图片粘贴/拖拽/插入。
- Markdown 兼容：用 Markdown 快捷输入与 HTML/ProseMirror 文档作为内部存储格式，后续增加 Markdown 导入/导出。
- 数据层：SQLite + WAL + FTS5，便签列表只加载摘要，正文按需加载。
- 性能策略：侧栏虚拟列表、输入防抖保存、图片写入前压缩、搜索走 FTS。
- 可靠性：本地持久化、软删除、版本号、后续可扩展同步冲突处理。

## 当前实现差距

- 图片原先使用 `blob:` 临时 URL，应用重启后图片会失效；已改为压缩后的 data URL 持久保存。
- 正文存储为 HTML，但标题、预览和搜索摘要原先按 Markdown/裸字符串处理，显示和搜索质量会偏差；已补纯文本提取。
- FTS 查询原先直接使用用户输入，特殊字符可能导致搜索命令失败；已做查询归一化。
- 延迟保存原先可能在切换便签后把旧保存结果写回当前状态；已限制只更新仍处于激活状态的便签。
- 虚拟列表高度原先用 `window.innerHeight` 一次性计算，窗口大小变化后可能错位；已改为监听容器尺寸。

## 后续优先级

1. 增加 Markdown 导入/导出，以及复制为 Markdown。
2. 将图片从 data URL 迁移为本地附件目录，正文只保存附件引用，降低数据库膨胀风险。
3. 增加自动保存状态、失败重试和用户可见错误提示。
4. 增加回收站、撤销删除和历史版本恢复。
5. 为中文搜索接入更合适的分词或维护纯文本索引列。
6. 增加端到端测试：创建、编辑、插图、搜索、重启后恢复。
